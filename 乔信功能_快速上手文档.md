## 3. "乔信" 核心功能详解

> **"乔信"就是 `chatWithCust` + `chatWithCustDetail` 两个页面**，实现客户之间的即时通讯（类似微信聊天）。

### 3.1 文件清单

```
pages/module_others/pages/
├── chatWithCust/          # 会话列表页（4个Tab）
│   ├── chatWithCust.js    # → 业务逻辑
│   ├── chatWithCust.wxml  # → 界面
│   ├── chatWithCust.wxss  # → 样式
│   └── chatWithCust.json  # → 页面配置
├── chatWithCustDetail/    # 聊天详情页
│   ├── chatWithCustDetail.js
│   ├── chatWithCustDetail.wxml
│   ├── chatWithCustDetail.wxss
│   └── chatWithCustDetail.json
└── utils/
    └── wechat.format.atim.js  # 仿微信时间格式化
```

### 3.2 页面结构 & 跳转关系

```
chatWithCust（会话列表）
  ├── Tab 0: 叮客（会话列表）  ← 展示所有聊天会话，点击进入 chatWithCustDetail
  ├── Tab 1: 通讯录            ← 占位，未实现
  ├── Tab 2: 叮友（绑客）       ← 占位，未实现
  └── Tab 3: 我                ← 设置头像/昵称/手机号

点击会话 → wx.navigateTo → chatWithCustDetail（聊天详情）
  传参：custInfo=JSON.stringify(item)
  包含字段：customer_id, customer_name, avatar_url, shop_id 等
```

### 3.3 chatWithCust — 会话列表页 核心逻辑

**文件**: `pages/module_others/pages/chatWithCust/chatWithCust.js`

**关键变量** (`data`)：

| 变量名 | 含义 | 注意 |
|--------|------|------|
| `msgList` | 会话列表数组 | 每个元素包含 `customer_id`, `customer_name`, `avatar_url`, `lstWechatMsgRecord`（最新消息数组） |
| `total_is_not_read_count` | 总未读消息数 | 用于底部 "叮客" Tab 红点 |
| `pageIndex` | Tab 索引（0-3） | 0=叮客, 1=通讯录, 2=叮友, 3=我 |
| `phone`, `nickname`, `avatarUrl` | "我"Tab 的用户信息 | — |

**核心函数**：

| 函数 | 作用 | 调用时机 |
|------|------|---------|
| `initMsg()` | 拉取会话列表 | `onLoad` 时调用 |
| `selectWechatMsgRecord_Last(...)` | 刷新单个会话的最后一条消息 | 收到推送时被调用 |
| `chatWithCustDetail(e)` | 跳转聊天详情 | 点击会话条目 |
| `onChooseAvatar(e)` | 选择头像 | "我"Tab 点击头像 |
| `getPhoneNumber(e)` | 获取微信手机号 | "我"Tab 点击获取手机号 |
| `saveNickName(e)` | 保存昵称 | "我"Tab 设置昵称 |
| `upLoadAvatarUrl()` | 上传头像到 OSS | 选择头像后自动调用 |
| `updateCustomerByOpenId(data, url)` | 通用更新用户资料 | 被头像/昵称/手机号保存复用 |
| `selectCustomerInfByOpenIdNew()` | 查询用户最新资料 | 初始化 + 资料更新成功回调 |
| `pageChangeEvent(e)` | Tab 切换 | 底部四个 Tab 点击 |

**会话列表拉取流程**:
```
1. onLoad → initMsg()
2. 调接口 selectWeChatMsgRecord_QGQC_Customer（参数：shop_id, bind_person_id）
3. 拿到数据后：用 at.compare() 按 send_time 排序
4. 遍历计算总未读数和每条消息的格式化时间
5. setData 更新视图
6. 过滤：customer_name == '未知' 的会话不显示
```

### 3.4 chatWithCustDetail — 聊天详情页 核心逻辑

**文件**: `pages/module_others/pages/chatWithCustDetail/chatWithCustDetail.js`

**关键变量** (`data`)：

| 变量名 | 含义 | 注意 |
|--------|------|------|
| `custInfo` | 对方用户信息 | 从 onLoad 的 JSON.parse(options.custInfo) 获取 |
| `customerInf` | 当前用户信息 | = `app.globalData.customerInf` |
| `recordList` | 消息列表数组 | 核心数据，每条消息包含 id/sender_id/msg_type/msg/duration |
| `msg` | 输入框文本 | 双向绑定 |
| `pageId` | scroll-into-view 目标 | `"page_" + 最后一条消息.id`，用于自动滚动 |
| `keyboardUp` / `utilUp` | 键盘/工具面板状态 | 控制键盘和更多面板的收起/展开 |
| `isOldLoading` | 是否正在加载旧消息 | 防重复请求 |

**消息类型 `msg_type`**：

| 值 | 类型 | 说明 |
|----|------|------|
| 1 | 文字 | msg 字段为文本 |
| 2 | 图片 | msg 字段为 OSS URL |
| 3 | 语音 | msg 字段为 OSS URL，duration 为秒数 |
| 4 | 视频 | msg 字段为 OSS URL |

**聊天类型 `wechat_type`**：

| 值 | 说明 |
|----|------|
| 1 | 店员 → 客户 |
| 2 | 客户 → 店员 |
| **3** | **客户 → 客户（乔信专用）** |

**核心函数**：

| 函数 | 作用 | 调用时机 |
|------|------|---------|
| `getOldData()` | 拉取历史消息（向上翻页） | onLoad + scrollToUpper（滚动到顶） |
| `getNewData()` | 拉取新消息 | 发送消息后 + 收到推送后 |
| `sendMsgExec(msg_type, msg, duration)` | 发送消息核心 | 发送文字/图片/视频/语音 |
| `sendRabbitMqMsg(...)` | 推送通知给接收方 | 发送成功后调用 |
| `chatWithCust_Last(...)` | 通知列表页刷新 | 每次获取数据后调用 |
| `uploadImage(filePath)` | 上传图片到 OSS → 发消息 | 选择图片后 |
| `uploadVideo(filePath)` | 上传视频到 OSS → 发消息 | 选择视频后 |
| `uploadAudio(filePath, duration)` | 上传语音到 OSS → 发消息 | 录音结束后 |
| `chooseMedia(e)` | 选择图片/视频 | 点击 + 面板的图片/视频按钮 |
| `recordStart(e)` / `recordEnd(e)` | 长按录音 | 切换为语音模式后长按 |
| `showBigPhoto(e)` | 预览大图 | 点击图片消息 |
| `playAudio(e)` | 播放语音 | 点击语音消息 |
| `changeKeyboardType()` | 切换键盘/语音输入 | 点击左下角图标 |
| `getKeyboardHeight(e)` | 获取键盘高度 | 键盘弹起事件 |
| `showKeyboardDown(e)` | 收起键盘 | 点击聊天区域 |
| `utilChange(e)` | 展开/收起 + 面板 | 点击 + 号 |
| `scrollToUpper(e)` | 到达顶部加载更多 | scroll-view 事件 |

**消息发送完整流程**：
```
1. 用户操作（输入文字/选择图片/长按录音/选择视频）
2. 如果是文件 → uploadFile 到 OSS → 拿到 URL
3. 调 sendMsgExec(msg_type, msg, duration)
   ├── 构造 data: {shop_id, msg_type, msg, duration, wechat_type:3, sender_id, receiver_id}
   ├── POST → insertWechatMsgRecord 写入数据库
   ├── 成功 → getNewData() 拉取新消息刷新界面
   ├── 成功 → sendRabbitMqMsg() 推送通知给接收方
   └── 如果是文字 → 清空输入框
```

**历史消息分页（getOldData）**：
```
1. 取当前 recordList[0].id 作为 max_id（首次为 0 = int 最大值）
2. GET → selectWechatMsgRecord_StaffToCustomer_Old
   参数: max_id, page_size:20, customer_id, shop_id, staff_id(=对方ID)
3. 返回数据 concat 到 recordList 前面
4. 设置 pageId 为 "page_" + 新数据最后一条的 id（滚动定位）
```

**新消息拉取（getNewData）**：
```
1. 取当前 recordList[last].id 作为 min_id
2. GET → selectWechatMsgRecord_StaffToCustomer_New
   参数: min_id, page_size:20, customer_id, shop_id, staff_id
3. 返回数据 concat 到 recordList 后面
4. 调 chatWithCust_Last() 通知列表页刷新
```

### 3.5 消息气泡 UI 逻辑

```
WXML 中通过 item.sender_id != customerInf.id 判断：
  - sender_id != customerInf.id → 对方消息 → 左侧白色气泡 + 对方头像
  - sender_id == customerInf.id → 自己消息 → 右侧绿色气泡(#89d961) + 自己头像

消息类型渲染：
  - msg_type=1（文字） → 直接显示文本
  - msg_type=2（图片） → image 标签 + 点击预览大图
  - msg_type=3（语音） → 语音图标 + duration 秒数，气泡宽度 = (duration%61+5)*2 rpx
  - msg_type=4（视频） → video 标签
```

## 4. 实时消息机制（核心架构）

> 乔信不用轮询，走 **RabbitMQ + WebSocket（STOMP 协议）** 实时推送。

### 4.1 架构全景

```
┌──────────────┐   发送消息    ┌──────────────┐   POST API    ┌──────────────┐
│  发送方小程序  │ ──────────→ │  Java 后台     │ ────────────→ │   MySQL       │
│              │             │                │              │              │
└──────────────┘             │  insertWechat  │              └──────────────┘
                             │  MsgRecord     │
                             │                │  SendRabbitMqMsg
                             │  SendRabbitMq  │ ────────────→ ┌──────────────┐
                             │  Msg API       │              │  RabbitMQ     │
                             └──────────────┘              └──────┬───────┘
                                                                  │
                                                    推送消息到接收方队列
                                                    队列名: {hj}_qgqc_{receiver_id}
                                                                  │
                                                          ┌───────┴───────┐
                                                          │  STOMP/WS      │
                                                          │  (RMQV3.js)    │
                                                          └───────┬───────┘
                                                                  │
                              ┌───────────────────────────────────┤
                              ▼                                   ▼
                    ┌──────────────────┐              ┌──────────────────┐
                    │ 接收方小程序       │              │ 接收方小程序       │
                    │ WebSocket 收到     │              │ (另一用户)         │
                    └──────────────────┘              └──────────────────┘
```

### 4.2 WebSocket 初始化（RMQV3.js）

```
app.js onShow → 检查 RMQOpenStatus
  → rmq.initSocket(customerInf.userCode)
    → wx.connectSocket('wss://{mqtt_url}/ws')
    → STOMP over WebSocket
    → 订阅目标队列: "{hj}qgqc_{userCode}"（如 "real_qgqc_10246"）
    → 收到消息 → app.chatWithCustDetail_GetNewMsg(msg)
```

### 4.3 消息路由分发（app.js 约 3000-3140 行）

```javascript
// 全局消息处理函数（在 app.js 中定义）
chatWithCustDetail_GetNewMsg(chatObj) {
    // 1. 获取当前页面栈
    // 2. 如果当前页面是 chatWithCustDetail → curPage.getNewData()
    // 3. 如果当前页面是 chatWithCust → curPage.selectWechatMsgRecord_Last(...)
    // 4. 如果上一页是 chatWithCust → prePage.selectWechatMsgRecord_Last(...)
    // 支持：同时更新聊天详情页 + 会话列表页
}

chatwithcrs_GetNewMsg(chatObj) { /* 商城客服聊天推送 */ }
chatwithdingke_GetNewMsg(chatObj) { /* 叮客聊天推送 */ }
```

RMQV3.js 中根据 `msg.type` 分发：
- `type == 'wechat'` → `app.chatWithCustDetail_GetNewMsg()` — 乔信
- `type == 'mall_csr_msg_record'` → `app.chatwithcrs_GetNewMsg()` — 商城客服
- `type == 'ding_ke_msg_record'` → `app.chatwithdingke_GetNewMsg()` — 叮客
- `type == 'refresh'` → `app.globalData.RMQRefreshCB()` — 通用刷新
- `tg == 'bz'` / `tg == 'fw'` → 呼叫部长/服务员响应

## 7. 变量名对照表（避坑指南）

> 这个项目很老，变量名有些奇怪，以下是常见的让你困惑的命名：

| 代码中的变量名 | 实际含义 | 注释 |
|---------------|---------|------|
| `custInfo` / `cust` | 对方客户（聊天对象） | Customer 的缩写，不是"当前用户" |
| `customerInf` | 当前登录用户信息 | `app.globalData.customerInf` |
| `bind_person_id` | 当前用户ID | 即 `customerInf.id`，命名历史遗留 |
| `staff_id` | 在乔信中是**对方客户ID** | 原来接口为店员→客户设计，乔信复用了该接口 |
| `customer_id` | 在乔信接口中有时是**当前用户ID** | 需根据具体接口判断 |
| `receiver_id` / `sender_id` | 接收方/发送方的客户ID | 乔信中 sender = 当前用户, receiver = 对方 |
| `recordList` | 聊天消息列表 | 每条消息叫 record |
| `msgList` | 会话列表 | 每个会话叫 msg |
| `is_not_read_count` | 未读消息数量 | 命名奇怪但不用改 |
| `lstWechatMsgRecord` | 最新一条消息的数据（数组） | 列表接口返回嵌套数组，取 [0] 使用 |
| `insert_customer` | 是否第一次发送消息 | 1=是，发送后变为 0 |
| `caustomerId` | customerId 的拼写错误 | app.js 全局变量，注意不是 `customerId` |
| `hj` | 环境标识 | "test"或"real"，用于 RabbitMQ 队列前缀 |
| `RMQ` | RabbitMQ | 整个项目消息推送的缩写 |

**最重要的混淆**：
- `staff_id` 参数在乔信（客户→客户）场景下传的是**对方客户ID**，不是店员ID！
- 因为 `selectWechatMsgRecord_StaffToCustomer_*` 接口原本为"店员→客户"场景设计，乔信复用时把"对方客户"当作 staff_id 传入。

## 8. 常见增删改查场景操作指南

### 8.1 新增一个消息类型（如"位置消息"）

需要修改的文件：
