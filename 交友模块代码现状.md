# 交友/脱单模块代码现状

> 记录时间：2026-06-22  
> 项目：qgqc-low-master（微信小程序前端）

---

## 本次定位：脱单套餐页面

- 页面注册位置：`app.json` 的 `pages/module_discount` 分包，包含 `pages/loveSet/loveSet`、`pages/buyLoveSet/buyLoveSet`、`pages/chooseLove/chooseLove`、`pages/activityPage/activityPage`。
- 交友入口页：`pages/module_discount/pages/loveSet/`，负责填写/校验姓名、电话、年龄、性别，并跳转套餐列表。
- 脱单套餐列表页：`pages/module_discount/pages/buyLoveSet/`，通过 `app.globalData.getSetMealByStartUsing3` 拉取脱单活动套餐。
- 套餐详情页：`pages/module_discount/pages/Package_details/`，从套餐列表进入时会带 `isFromBuyLoveSet=true` 和 `customerInf` 参数。
- 是否为脱单/交友套餐的核心判断字段：`setMeal[0].startUsing3 == 1`。
- 常见访问链路：`loveSet -> buyLoveSet -> Package_details -> order -> chooseLove`。

---

## 一、业务流程

```
loveSet（填写个人信息 + 检查注册/关注公众号）
  ↓ toBuyLoveSet()
buyLoveSet（选择活动套餐列表）
  ↓ toDetail()
Package_details（套餐详情）
  ↓ toOrder()
order（下单 → 微信支付 → 写 apply_combo + ticket_orderInfo）
  ↓ 支付成功后跳转
chooseLove（双向选择配对）
  ↓ 选择 / 同意 / 拒绝 / 结束
结束
```

**判断是否为交友活动套餐的字段：** `setMeal[0].startUsing3 == 1`

## 三、各页面关键逻辑

### 3.1 loveSet.js

- `onShow` 时从 `app.globalData.customerInf` 读取用户信息，计算年龄，修正性别默认值
- `getCustomerInfo()` → POST `/selectCustomerInfByOpenId` 获取用户信息
- `isFocus()` → 检查是否关注"切瓜切菜"公众号（固定正式库地址 `mb.fsmbdlkj.com`），未关注则弹窗引导
- `showRegister()` → 未注册时展示注册提示组件（`isRegister: true`）
- `toBuyLoveSet()` → 校验姓名/电话/年龄完整性后，携带 `customerInf` 跳转 buyLoveSet
- 支付成功后回到本页，从 `app.globalData.loveSetMeal` 读取已报名套餐信息展示（`isSet: true`）

关键 data 字段：
```js
realName, phone, age, genderIndex  // 用户信息
isSet         // 是否已报名套餐
loveSetMeal   // 已报名的套餐信息（支付后回显）
isRegister    // 是否显示注册提示
```

---

### 3.2 buyLoveSet.js

- `onLoad` 接收 `customerInf` 参数
- `selectAllSetMeal()` → GET `getSetMealByStartUsing3` 拉取活动套餐列表，分页累加
- `toDetail()` → 跳转 Package_details，传参：`setMealID`、`ruleID`、`isFromBuyLoveSet=true`、`customerInf`

WXML 中展示的套餐字段：
```
manPrice          男士价格/人
ladyPrice         女士价格/人
commission_manPrice   男士分销返佣
commission_ladyPrice  女士分销返佣
activityDate      活动用餐日期
```

---

### 3.3 order.js（脱单套餐相关部分）

**价格计算（按性别）：**
```js
actualSum = num * (gender == '女' ? ladyPrice : manPrice)
```

**用餐时间处理：**
- 优先用 `setMeal[0].activityDate.time`（套餐固定时间）
- 没有时 用用户自选 `usingDate`（格式化为 `Y-M-D H:m:s`）

**支付成功后调用 `addApplyCombo`，POST 字段：**
```js
{
  customer_open_id,
  customer_id,
  real_name,
  sex,              // 1=男 2=女
  age,
  phone,
  job: "",
  address,          // 来自 customer.city
  dinner_time,      // 格式：Y-M-D H:m:s
  setMealID,
  combo_name,
  combo_price,      // 实际支付金额
  union_id,
  shop_id,
  county_id: 0,
  customer_remark,
  ticket_order_inf_id  // 关联 ticket_orderInfo 表
}
```

**分销返佣：**
```js
percent_total = loveInviterOpenid != openid
  ? (gender == '女' ? commission_ladyPrice : commission_manPrice)
  : 0
```
- `app.globalData.loveInviterOpenid` 存储邀请人 openid，用于判断是否走分销

**支付成功后写全局：**
```js
app.globalData.loveSetMeal = {
  setMealName, ladyPrice, manPrice,
  usingDate,      // 格式化后的中文日期
  customerInf1    // 报名用户信息快照
}
```

**注意：** `startUsing3 == 3` 分支逻辑与 `== 1` 基本相同，但不调用 `addApplyCombo`，只走普通套餐订单流程。

### 3.4 chooseLove.js

**onLoad 时并行发两个请求：**
1. `selectApplyComboByCustomerId` → 获取同套餐、同用餐时间的所有报名用户（待我选的人），过滤掉没有 `distribute_num` 的（未被分配编号不显示）
2. `getChoiceList` → 获取"选了我"和"我选了谁"的列表

**核心操作：**

| 操作 | 方法 | API | 关键参数 |
|------|------|-----|----------|
| 发起选择 | `needChooseConfirm()` | `addChoose` POST | `apply_combo_me_uid`、`apply_combo_other_uid`、`me_num`、`other_num`、`setMealID`、`dinner_time` |
| 同意/拒绝 | `radioChange()` | `agreeOrRefuse` | `agreeStatus`（1=同意 2=拒绝）、`uid` |
| 我方结束 | `meEnd()` | `beEnd` | `index:1`、`meUid`、`uid` |
| 对方结束 | `beEnd()` | `beEnd` | `index:2`、`otherUid`、`uid` |

**`addChoose` 返回 code 含义：**
```
1 → 成功，刷新列表
2 → 对方正在谈话中
3 → 数量超出，请等待
4 → 已发送过申请
```

**前置检查：** `distribute_num` 为 0 时禁止发起选择，提示"请等待分配编号"

## 六、注意事项 / 已知问题

1. **`loveSet.js` `onLoad` 为空**：`onLoad` 只有 `console.log`，实际初始化逻辑全在 `onShow` 里，首次进入依赖 `app.globalData` 已有数据。

2. **公众号关注检查写死正式库地址**：`loveSet.js:71` 直接硬编码 `https://mb.fsmbdlkj.com/wxpublic/WXPublic/getList`，本地调试时不受 `serverConfig` 控制。

3. **`chooseLove.js` 中 `beEnd` 方法索引问题**：`beEnd()` 第 163 行用 `that.data.ChooseList[e.currentTarget.dataset.id].uid`，但当前 `ChooseList` 未在 `onLoad` 中初始化，只在 `onShow` 里赋值，`onLoad` 时可能为空。

4. **`startUsing3 == 3` 分支**：`order.js` 中有 `startUsing3 == 3` 的分支（第 1011 行），逻辑与 `== 1` 相似但不调 `addApplyCombo`，该类型套餐的完整流程尚不明确。

5. **分销返佣**：邀请人 openid 存在 `app.globalData.loveInviterOpenid`，来源页面未在此模块内，需排查落地页/分享入口。
